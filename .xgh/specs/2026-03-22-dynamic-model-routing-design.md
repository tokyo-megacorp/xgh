# Dynamic Model Routing

> Automatically select the best agent + model + effort for each dispatched task, learned from real usage data per-repo and per-developer.

## Problem

xgh dispatch skills (codex, gemini, opencode) support model and effort flags, but selection is fully manual. Users must know which model works best for which task type across providers. This knowledge exists implicitly in session history but isn't captured or reused.

## Design Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Architecture | Router skill (Approach C) | Single entry point, dispatch skills stay unchanged, most testable |
| Model catalog | None — discovered from usage | No static config to maintain, fully emergent |
| Profile storage | `.xgh/model-profiles.yaml` (gitignored) | Per-repo, per-developer, grows organically |
| Cold start | CLI defaults | Zero friction, starts collecting data immediately |
| Quality signal | Binary `accepted: true/false` | Simple enough to ship, granular scoring can come later |
| Complexity axis | Dropped | Archetype alone drives lookup — one axis, not two |

## Components

### 1. `agents.yaml` — No Changes

The agent registry stays as-is. No `models` block. The router discovers available models from usage observations.

### 2. `/xgh-dispatch` Router Skill

New skill at `skills/dispatch/dispatch.md`.

**Responsibility:** Classify task, lookup profile data, select agent+model+effort, invoke the appropriate dispatch skill with flags injected.

**Input:** User provides a task description and optional overrides.

```
/xgh-dispatch "write unit tests for the auth module"
/xgh-dispatch --agent codex "fix the typo in README"
/xgh-dispatch --model opus "design the new API"
```

**Step 1 — Classify:**

The LLM reads the task description and determines the archetype:

| Archetype | Signals |
|-----------|---------|
| `brainstorming` | Creative exploration, specs, ideation |
| `planning` | Implementation plans, task breakdown |
| `implementation` | Writing code, feature work |
| `code-review` | Reviewing PRs, quality checks |
| `debugging` | Investigation, root cause analysis |
| `refactoring` | Restructuring existing code |
| `documentation` | Writing docs, comments |
| `quick-task` | Typo fixes, config changes, one-liners |

**Step 2 — Lookup:**

Read `.xgh/model-profiles.yaml`. Filter observations by archetype. Among entries where `accepted: true`, pick the agent+model+effort with the most observations. If tied, prefer the one used most recently.

**Step 3 — Select:**

Output a tuple: `{agent, model, effort}`. If no profile data exists for this archetype (cold start), use CLI defaults — invoke the dispatch skill without `-m` or `--effort` flags.

**Step 4 — Dispatch:**

Invoke the appropriate dispatch skill with agent-specific flags:

```
/xgh-codex exec -m <model> --effort <effort> "<task>"
/xgh-gemini exec -m <model> --effort <effort> "<task>"
/xgh-opencode exec --model <provider>/<model> "<task>"
```

Note: OpenCode has no effort flag — only model is passed. Codex and Gemini support both model and effort. The router must respect each agent's supported flags (defined in the dispatch skill, not here).

**Override behavior:**

| User says | Router behavior |
|-----------|----------------|
| `--model X` | Skip classification, infer agent from model prefix (`gpt-*` → codex, `gemini-*` → gemini, `anthropic/*` or `openai/*` → opencode). Require `--agent` if ambiguous. |
| `--agent codex` | Force codex, router picks model from profile |
| `--model X --agent codex` | Fully manual, router just passes through |
| `/xgh-codex exec "..."` | Bypass router entirely (direct dispatch) |

### 3. `.xgh/model-profiles.yaml` — Observation Store

Gitignored. Created on first observation. Structure:

```yaml
# Auto-generated by xgh dispatch curate step
observations:
  - agent: codex
    model: gpt-5.4-mini
    effort: low
    archetype: quick-task
    accepted: true
    ts: 2026-03-22T14:00:00Z
  - agent: codex
    model: gpt-5.4
    effort: high
    archetype: implementation
    accepted: true
    ts: 2026-03-22T15:30:00Z
```

Each observation is one dispatch result. No aggregation — the router computes rankings at lookup time from raw observations.

### 4. Curate Extension — Learning Loop

Each dispatch skill's existing "Step 5: Curate" is extended to write one observation after every dispatch completes.

**Fields recorded:**

| Field | Source |
|-------|--------|
| `agent` | Which skill dispatched (codex/gemini/opencode) |
| `model` | The `-m` flag used, or "default" if none |
| `effort` | The `--effort` level used, or "default" if none |
| `archetype` | Set by router, or "unknown" if direct dispatch |
| `accepted` | Did the user accept the result? (merged worktree, continued work, no re-dispatch = true) |
| `ts` | ISO timestamp |

**Acceptance inference (passive, no prompts):**

- Worktree merged → `accepted: true`
- User continued to next task → `accepted: true`
- User re-dispatched same task → `accepted: false`
- User discarded worktree → `accepted: false`

Direct dispatches (bypassing router) still record observations with `archetype: unknown`. These inform model quality but don't feed archetype-based routing.

## Test Strategy

### `test-dispatch-router.sh`

- Router skill file exists at `skills/dispatch/dispatch.md`
- Skill contains all 8 archetype definitions
- Skill references `.xgh/model-profiles.yaml`
- Skill references override flags (`--model`, `--agent`)
- Skill references fallback to CLI defaults

### `test-model-profiles.sh`

- `.gitignore` includes `model-profiles.yaml`
- Observation schema documented in skill (agent, model, effort, archetype, accepted, ts)

### `test-skills.sh` extensions

- All dispatch skills (codex, gemini, opencode) reference curate observation write
- Router skill registered in appropriate indexes

### Not tested statically

- Classification accuracy (behavioral — validated by dogfooding)
- Quality signal reliability (behavioral — tuned from usage)
- Profile lookup ranking (behavioral — adjusted as data accumulates)

## Dogfooding Plan

Use `/xgh-dispatch` for all task dispatch during implementation of this feature itself. Track which models get selected, whether the cold-start defaults are reasonable, and whether the acceptance signals are accurate. Adjust the classification prompt and acceptance heuristics based on real observations.
